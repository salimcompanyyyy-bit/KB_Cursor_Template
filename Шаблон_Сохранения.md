# Настройка ИИ-агента проекта — полный текст для нового Cursor-проекта

**Это главный файл дисциплины сохранения.** После любой существенной правки кода агент обязан довести порцию до коммита по этому документу. Краткий указатель в репозитории: **`context/КОНТЕКСТ_СОХРАНЕНИЕ.md`**. Правило Cursor с `alwaysApply: true`: **`.cursor/rules/save-on-code-change.mdc`**.

Скопируйте этот файл в новый проект или вставьте фрагменты в `.cursor/rules/*.mdc` и попросите агента всегда следовать этому документу.

---

## 1. Общая модель работы агента

Агент работает в Cursor над конкретным проектом. У него есть:

1. **Обязательные правила** проекта в `.cursor/rules/*.mdc` — их агент читает **в начале каждой существенной задачи**, даже если «помнит» их из прошлой сессии.
2. **Контекстные файлы** в `context/`:
   - `context/КОНТЕКСТ_АГЕНТА.md` — рабочая память агента (что уже сделано, какие решения приняты, текущее поведение системы).
   - `context/КОНТЕКСТ_ИДЕЙ.md` — бэклог идей/улучшений (только нереализованное; сделанное — удаляется).
   - `context/КОНТЕКСТ_ПОЛЬЗОВАТЕЛЯ.md` — постоянные предпочтения пользователя (например, мультивыбор в вопросах).
3. **Файл версии** в корне проекта: **`VERSION`** — одна строка вида `MAJOR.MINOR.PATCH`.
4. **Git-история**: каждый завершённый шаг = один коммит по строгому формату (см. ниже).

Главный принцип: **«изменено → проверено → закоммичено»**. Нельзя говорить «готово», если коммит не создан.

## 2. Структура файлов настройки

```
.cursor/
  rules/
    mandatory-rules-read-first.mdc    # принудительное чтение правил
    save-on-code-change.mdc           # ОБЯЗАТЕЛЬНО: VERSION + git commit после правок кода
    project-context-template.mdc       # единое правило проекта (версии, коммиты, уточнения)
    mib-context.mdc                    # в KB_MIB_PARSER: доменные правила mib.uz (опционально в других проектах)
context/
  КОНТЕКСТ_СОХРАНЕНИЕ.md              # указатель на этот шаблон + чеклист для агента
  КОНТЕКСТ_АГЕНТА.md                   # рабочая память агента (lean; история → АРХИВ/)
  КОНТЕКСТ_ИДЕЙ.md                    # бэклог идей (только открытое)
  КОНТЕКСТ_ПОЛЬЗОВАТЕЛЯ.md             # предпочтения пользователя
  АРХИВ/                               # готовые спеки, модель Excel, RBAC, changelog
VERSION                              # например 2.0.0, 2.0.1, 2.0.2 ...
```

У файлов `.mdc` в начале задаётся фронт-маттер:

```yaml
---
description: краткое описание правила
alwaysApply: true
---
```

`alwaysApply: true` означает, что Cursor обязан подкладывать это правило агенту **всегда**.

---

## 3. Правило «Mandatory Rules Read First»

Содержание `.cursor/rules/mandatory-rules-read-first.mdc`:

- Перед любой существенной работой агент **обязан** сначала прочитать все `.mdc` в `.cursor/rules/`.
- Только после этого — анализировать, править код, запускать команды, коммитить.
- При появлении новых правил в сессии — перечитать их перед следующей существенной задачей.
- Нельзя пропускать чтение правил, даже если агент «помнит» их.
- Нельзя начинать реализацию до чтения правил.
- При конфликте инструкций с правилами проекта: **явные указания пользователя в текущем чате, переопределяющие правило, имеют приоритет** (см. примечание в конце файла). Иначе агент следует правилам проекта и предлагает безопасный путь.

## 4. Единое правило проекта (главное)

Содержание `.cursor/rules/project-context-template.mdc` обычно разбито на блоки **0–7** (уточнения, источник правды, версии, git, дисциплина, словарь типов, обновление контекста), согласованные с подразделами **4.1–4.7** ниже.

### 4.1. Сначала уточнения, потом код

1. Если задача неоднозначна или не хватает данных — сначала задать вопросы пользователю.
2. Любая новая видимая функциональность (логи, тексты, CLI-флаги, заметное поведение) — сначала уточнение, потом реализация.
3. Если варианты ответа можно перечислить заранее — агент использует форму **Ask Question** (в Cursor это отдельный UI).
4. Пока ответы не получены, код по новой видимой функциональности **не** пишется.
5. Любые уточняющие вопросы — только через **Ask Question**.

### 4.2. Источник правды и границы изменений

1. Перед существенными изменениями **сначала** читаем `context/КОНТЕКСТ_АГЕНТА.md` и `context/КОНТЕКСТ_ПОЛЬЗОВАТЕЛЯ.md` (постоянные предпочтения).
2. **Дополнительно** читаем `context/КОНТЕКСТ_ИДЕЙ.md` при работе с бэклогом идей или планированием улучшений.
3. Перед тем как сказать пользователю «сделано», агент **сверяется с прочитанным контекстом** и не противоречит ему.
4. Работаем с **минимальным диффом**: меняем только то, что нужно для задачи.
5. При конфликте старых заметок и кода — источником правды считается **актуальный код + КОНТЕКСТ_АГЕНТА.md**.
6. Язык общения с пользователем — **русский**.

### 4.3. Версии проекта (обязательно)

1. Базовая версия задаётся в начале (например, `2.0.0`).
2. Версия хранится в корневом файле **`VERSION`** одной строкой: `MAJOR.MINOR.PATCH`.
3. При существенных изменениях версию **обновляет сам агент** в той же сессии:
   - **PATCH** — багфиксы и мелкие улучшения;
   - **MINOR** — новая функциональность и заметные улучшения;
   - **MAJOR** — **только** по явному согласованию с пользователем.
4. По умолчанию проект на ветке `2.0.x`:
   - после изменений повышается **только PATCH** (`2.0.1`, `2.0.2`, ...);
   - переход на `2.1.0` или выше — **только** по явному подтверждению пользователя.

Для нового проекта строку «2.0.x» можно заменить на свою (например, `1.0.x`), логика та же.

### 4.4. Git: когда и как коммитить

1. **Один коммит = одна законченная и проверяемая порция работы**.
2. Перед коммитом агент обязан:
   - проверить `git status`;
   - добавить только релевантные файлы `git add ...`;
   - убедиться, что нет секретных/лишних файлов в staged;
   - выполнить `git commit`.
3. **Не коммитим** в репозиторий как отслеживаемое содержимое: **файлы и пути, подпадающие под `.gitignore`**; локальные рабочие данные по умолчанию; секреты (`.env`, ключи). Сам файл **`.gitignore`** коммитится, когда меняются правила игнора.

4. **Формат коммита (строго):**
   - первая строка: `vX.Y.Z: <тип> - <кратко>` (версия **совпадает** с файлом `VERSION`);
   - `<тип>` — одно русское слово из фиксированного словаря (см. 4.6);
   - `<кратко>` — 2–4 слова;
   - **пустая строка**;
   - **нумерованный список из 1–3 пунктов** — что изменено.

5. `git push` — **только по явной просьбе** пользователя.

Пример правильного коммита:

```
v2.0.26: исправление - отключено автоудаление

1. Глобально отключено автоудаление сообщений через флаг AUTO_DELETE_ENABLED=False.
2. Выключены автоматические удаления в middleware, send_step и автоочистке.
3. Обновлены VERSION и КОНТЕКСТ_АГЕНТА под новое поведение.
```

Даже `git revert` оформляется по этому шаблону, не стандартным сообщением вида `Revert "..."`.

### 4.5. Дисциплина фиксации результата

1. Задача по коду/правилам доводится до **«изменено + проверено + закоммичено»**.
2. Нельзя писать «готово», если коммит **не** создан.
3. Если коммит сознательно не делается — агент **явно** пишет об этом в ответе.

### 4.6. Словарь типов коммитов

Разрешены **только** эти типы (одно слово):

| Тип               | Назначение |
|-------------------|------------|
| `добавление`      | что-то новое в проект |
| `исправление`     | багфикс, мелкая корректировка |
| `функция`         | заметная новая функциональность |
| `рефакторинг`    | изменение структуры без смены поведения |
| `контекст`        | context/\*, правила, документы без смены поведения кода |
| `база`            | БД, миграции, схема |
| `безопасность`    | секреты, доступ, жёсткая валидация |
| `производительность` | скорость, память, нагрузка |

### 4.7. Обновление контекста после изменений

1. Меняется поведение или workflow → обновить `context/КОНТЕКСТ_АГЕНТА.md`.
2. Нужны пользовательские предпочтения → `context/КОНТЕКСТ_ПОЛЬЗОВАТЕЛЯ.md`.
3. Идея реализована → удалить пункт из `context/КОНТЕКСТ_ИДЕЙ.md`.

## 5. Полный цикл задачи (алгоритм агента)

1. Прочитать все `.mdc` в `.cursor/rules/`.
2. Прочитать `КОНТЕКСТ_АГЕНТА.md`, `КОНТЕКСТ_ИДЕЙ.md`, `КОНТЕКСТ_ПОЛЬЗОВАТЕЛЯ.md`.
3. Если неясно — **Ask Question** (предпочтительно мультивыбор).
4. Минимальный дифф по коду.
5. При необходимости — проверки/линтер.
6. Поднять `VERSION` (обычно +1 PATCH).
7. Обновить контексты.
8. `git status` → `git add` → `git commit` по формату 4.4.
9. Проверить чистоту дерева.
10. Краткий ответ пользователю + хеш/заголовок коммита.

---

## 6. Что агент не делает без явной просьбы

- Не делает `git push`.
- Не поднимает `MAJOR`/`MINOR` без согласования.
- Не правит правила произвольно без запроса.
- Не коммитит `.env`, секреты, мусор из `.gitignore`.
- Не делает огромный рефакторинг «впридачу».

---

## 7. Общение и Ask Question

- Язык — русский.
- Уточнения пользователю — **только Ask Question**.
- **ВСЕГДА мультивыбор, без исключений** (требование пользователя **2026-05-26**, hard-rule): у **каждого** вопроса в JSON формы **обязательно** **`"allow_multiple": true`**. **Никаких** `false`, даже для «ровно один из взаимоисключающих» / «выберите одно» / «один основной режим» / «Другое». Если по логике задачи действительно нужен ровно один ответ — задавать **два последовательных вопроса** (или дописать в тексте вопроса «отметьте только нужное»), но `allow_multiple` остаётся `true`. Пользователь **должен** мочь отметить несколько несмежных пунктов в **любой** форме. Перед отправкой формы — проверить JSON: `allow_multiple: true` у **всех** вопросов.

> Перед задачами читай все `.cursor/rules/*.mdc` и `context/КОНТЕКСТ_*.md`. Уточнения только через Ask Question с мультивыбором по возможности. Минимальный дифф.
