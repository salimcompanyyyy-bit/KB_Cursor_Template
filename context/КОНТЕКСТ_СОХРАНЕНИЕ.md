# Сохранение изменений в репозитории (обязательно для агента)

**Этот файл — указатель.** Полные правила, формат коммита и словарь типов — в корне репозитория:

## [`Шаблон_Сохранения.md`](../Шаблон_Сохранения.md)

Cursor всегда подмешивает правило [`.cursor/rules/save-on-code-change.mdc`](../.cursor/rules/save-on-code-change.mdc) (`alwaysApply: true`).

---

## Принцип

**Изменено → проверено → закоммичено.**

После существенной правки кода агент **в той же сессии**:

1. **`VERSION`** — обычно +1 PATCH (см. файл в корне).
2. **`context/КОНТЕКСТ_АГЕНТА.md`** — если изменилось поведение или договорённости.
3. **`git add`** + **`git commit`** по формату из **`Шаблон_Сохранения.md`**.
4. Убедиться, что порция зафиксирована (`git status`).

**Push** — только если пользователь явно попросил.

**Исключение:** пользователь в **текущем чате** написал «без коммита» / «не коммить» — тогда коммит не делать, но **явно** сказать об этом в ответе.

**Не ждать** слов «закоммить» / «сохрани» — коммит входит в завершение порции (**`save-on-code-change.mdc`**).

### Частые ошибки (запрещено)

- Правки есть, **`git commit` нет** — в Git Graph пусто, пользователь считает, что работа потеряна.
- **`VERSION`** поднят (например 3.1.12), а в истории только **3.1.10** — пропущен коммит **3.1.11**.
- Ответ «готово» **без** строки `vX.Y.Z` и **хеша** последнего коммита.

---

## Формат коммита (кратко)

```
vX.Y.Z: <тип> - <кратко>

1. …
2. …
```

- `X.Y.Z` = строка из **`VERSION`** после правки.
- `<тип>` — одно слово из шаблона: `добавление`, `исправление`, `функция`, `рефакторинг`, `контекст`, `база`, `безопасность`, `производительность`.

Подробности — **`Шаблон_Сохранения.md`**, раздел про git.
